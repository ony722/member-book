## 1. はじめに

### 1.1 目的

紙のメンバーブック（会員名簿・冊子）をめくる体験をWeb上で再現し、会員（メンバー）を1人1ページ・文字とリンクのみで紹介する静的Webサイトを構築する。参照UX: ActiBook/CloudCircus基盤のフリップブックビューア（本格的な紙めくりアニメーション）。

### 1.2 範囲（スコープ）

- **対象**: [docs/scope_table.md](./scope_table.md) の IN SCOPE 項目（フリップブックUI、1人1ページの会員紹介、ページ送り操作、目次・表紙、レスポンシブ対応、静的JSONデータ管理、全画面表示）
- **対象外**: 管理画面、会員写真、印刷/PDF、シェア機能、認証、実データ入力（詳細は scope_table.md）

### 1.3 用語集

| 用語 | 定義 |
|---|---|
| フリップブック | 本のページをめくるようなアニメーションでコンテンツを閲覧するUI形式 |
| 会員ページ | 1人の会員（メンバー）を紹介する1ページ分のコンテンツ（テキスト＋リンク） |
| 目次ページ | 全会員の一覧から任意の会員ページへジャンプできるページ |
| 表紙ページ | サイトの最初に表示される扉ページ |
| StPageFlip / page-flip | 採用するページめくりOSSライブラリ（[research_v2.md](../research/research_v2.md) 参照） |

## 2. システム概要

会員ブックサイトは以下の3種のページで構成される静的サイトである。

```
[表紙ページ] → [目次ページ] → [会員ページ1] ⇄ [会員ページ2] ⇄ ... ⇄ [会員ページN]
                    ↑____________________________________________________|
                    （目次から任意の会員ページへ直接ジャンプ可能）
```

- 訪問者はブラウザでサイトURLにアクセスし、表紙→目次→各会員ページへとページをめくりながら閲覧する
- サーバーサイド処理・データベース・認証は存在せず、ビルド時に静的JSONを埋め込んだHTML/CSS/JSとして配信される

## 4. 運用サイクル

| サイクル | 内容 |
|---|---|
| 会員データ更新時（開発者向け） | `src/data/members.json` を編集 → PR作成・レビュー → `main`マージ → ビルド（`npm run build`）→ デプロイ（GitHub Actions経由でGitHub Pagesへ自動反映） |
| 会員データ更新時（非エンジニア向け・簡易手順） | GitHub Web UI上で`src/data/members.json`を直接開き「編集(鉛筆アイコン)」からJSONを修正しコミット（ブランチ作成→PR作成の画面もWeb UI上で完結）。ローカル環境・Gitコマンド操作は不要。手順は別途「非エンジニア向け更新ガイド」として用意する（Round1レビュー指摘F-O3対応。risk_register.md RISK-003参照） |
| 定期タスク | なし（外部API・課金・自動投稿が存在しないため定期監視は不要） |
| 障害時 | ビルド失敗時はGitHub Actionsのログで検知し、直前のデプロイ済み静的ファイルはそのまま配信され続ける（ロールバック不要な構成） |

## 7. 機能要件（FR-*）

MoSCoW集計: Must 8件（57%）／ Should 4件（29%）／ Could 2件（14%）／ Won't 0件 → Must比率60%以下を満たす。

### FR-SYS-001: 表紙ページを表示する

- **優先度**: Must
- **Given**: 訪問者がサイトのルートURLにアクセスする
- **When**: ページの初回読み込みが完了する
- **Then**:
  - フリップブックの最初のページとして表紙（サイトタイトルを含む）が表示される
  - 表紙には「目次を開く」ための操作導線（クリック/ページめくり操作）が存在する
- **例外**: JSONデータの読み込みに失敗した場合、表紙の代わりにエラーメッセージ（「会員データを読み込めませんでした」）を表示する

### FR-SYS-002: 目次ページを表示する

- **優先度**: Must
- **Given**: 訪問者が表紙ページから1ページ進める操作を行った
- **When**: 目次ページに遷移する
- **Then**:
  - 全会員の名前一覧がページ内に表示される（会員数が1ページに収まらない場合は目次を複数ページに分割する）
  - 各会員名はクリック/タップ可能なリンクとして表示される
- **例外**: 会員データが0件の場合、「会員情報は準備中です」と表示する

### FR-SYS-003: 会員紹介ページを1人1ページとして表示する

- **優先度**: Must
- **Given**: 会員データ（`members.json`）に1件以上のレコードがある
- **When**: 該当会員のページが表示される
- **Then**:
  - 1ページに1名分のテキスト（名前・紹介文等）と、リンクが1件以上ある場合はそのリンク一覧のみが表示される（画像・写真等の要素は含まない）
  - 会員の写真・画像は表示しない（スコープ外）
- **例外**:
  - 紹介文（`bio`）が空文字の場合は名前とリンクのみを表示し、レイアウト崩れを起こさない
  - リンク（`links`）が0件の場合はリンク欄自体を表示しない（FR-DATA-001参照、ACC-FR-SYS-003・E2E-06で境界値として検証）

### FR-SYS-004: めくり操作（クリック/ドラッグ）でページ遷移する

- **優先度**: Must
- **Given**: フリップブックが表示されている
- **When**: 訪問者がページ端をクリックする、またはページをドラッグする
- **Then**:
  - 紙をめくるようなアニメーション（`page-flip`ライブラリ）と共に前後のページへ遷移する
  - アニメーション完了後、新しいページの内容が操作可能になる
- **例外**: 最終ページで次へめくる操作をした場合、何も起こらない（ループしない）

### FR-SYS-007: 目次から任意の会員ページへ直接ジャンプできる

- **優先度**: Must
- **Given**: 訪問者が目次ページを表示している
- **When**: 訪問者が特定の会員名をクリックする
- **Then**: フリップブックが該当会員のページまで自動的にページ送り（アニメーション付き）される
- **例外**: 存在しない会員IDが指定された場合、目次ページに留まりエラーは表示しない（内部リンクのみのため発生しない想定）

### FR-DATA-001: 静的JSONから会員データを読み込み表示する

- **優先度**: Must
- **Given**: `src/data/members.json` に会員データが定義されている
- **When**: サイトがビルド・読み込みされる
- **Then**:
  - 各会員レコードは `id`（一意）・`name`（必須）・`bio`（紹介文、空文字許容）・`links`（0件以上のリンク配列、各リンクは `label` と `url` を持つ）を持つ
  - JSONのパースに失敗した場合、ビルドが失敗し明確なエラーメッセージがCIログに出力される
- **例外**: `links` が0件の会員ページはリンク欄を表示しない

### FR-EXT-001: 会員ページ内のリンクをクリックすると新しいタブで外部サイトを開く

- **優先度**: Must
- **Given**: 会員ページにリンクが1件以上表示されている
- **When**: 訪問者がリンクをクリックする
- **Then**: `target="_blank" rel="noopener noreferrer"` で新しいタブが開き、元のフリップブックページはそのまま維持される
- **例外**: `url` が `http://` または `https://` で始まらない値の場合、ビルド時にバリデーションエラーとする

### FR-SYS-010: レスポンシブ対応（PC・タブレット・スマホ）

- **優先度**: Must
- **Given**: 訪問者が任意の画面幅（320px〜1920px想定）のデバイスでアクセスする
- **When**: ページが表示される
- **Then**: フリップブックの表示サイズが画面幅に応じて自動調整され、横スクロールが発生しない
- **例外**: 画面幅が320px未満の極端に小さいデバイスはサポート対象外とする（NFR-COMP-001参照）

### FR-SYS-005: 「次へ」「前へ」ボタンでページ送りができる

- **優先度**: Should
- **Given**: フリップブックが表示されている
- **When**: 訪問者が画面上の「次へ」または「前へ」ボタンをクリックする
- **Then**: めくりアニメーション付きで隣接ページへ遷移する
- **例外**: 最初/最後のページではそれぞれ「前へ」/「次へ」ボタンを非活性表示にする

### FR-SYS-006: キーボードの左右矢印キーでページ送りができる

- **優先度**: Should
- **Given**: PCブラウザでフリップブックにフォーカスがある
- **When**: 訪問者が右矢印キー/左矢印キーを押下する
- **Then**: 次のページ/前のページへめくりアニメーション付きで遷移する
- **例外**: 入力フォーム等フォーカスがテキスト入力中の場合はキー操作を無視する（本サイトにはテキスト入力欄がないため実質常に有効）

### FR-SYS-008: 現在ページ番号と総ページ数を表示する

- **優先度**: Should
- **Given**: フリップブックが表示されている
- **When**: 任意のページが表示中である
- **Then**: 画面内に「現在ページ / 総ページ数」の形式（例: `5 / 42`）でページ位置が表示される

### FR-SYS-011: 各会員ページから表紙・目次への戻り導線がある

- **優先度**: Should
- **Given**: 訪問者が会員ページを閲覧している
- **When**: 訪問者が「目次へ戻る」導線をクリックする
- **Then**: 目次ページへアニメーション付きで遷移する

### FR-SYS-009: 全画面表示への切り替えができる

- **優先度**: Could
- **Given**: フリップブックが表示されている
- **When**: 訪問者が全画面表示ボタンをクリックする
- **Then**: ブラウザのFullscreen APIによりフリップブックが画面全体に拡大表示される
- **例外**: Fullscreen APIが未対応のブラウザではボタンを非表示にする

### FR-SYS-012: ページ読み込み中のローディング表示

- **優先度**: Could
- **Given**: 初回アクセスでJSONデータ・アセットの読み込みが完了していない
- **When**: 読み込み処理中である
- **Then**: 簡易ローディングインジケーターを表示し、完了後に自動的に表紙ページへ切り替える

## 8. 画面・操作フロー

### 8.1 画面一覧

| 画面ID | 画面名 | 概要 | 空状態文言 |
|---|---|---|---|
| SCR-001 | 表紙ページ | サイトタイトルを表示する扉ページ | データ0件時: 「会員情報は準備中です」 |
| SCR-002 | 目次ページ | 全会員名の一覧（会員数に応じて複数ページに分割） | 会員データ0件時: 「会員情報は準備中です」 |
| SCR-003 | 会員ページ | 1人1ページ（名前・紹介文・リンク） | 紹介文が空文字の場合はリンクのみ表示（欠落として扱わない） |

### 8.2 画面遷移図

```
SCR-001(表紙) --めくる/クリック--> SCR-002(目次)
SCR-002(目次) --会員名クリック--> SCR-003(該当会員ページ)
SCR-003(会員N) --次へ/めくる--> SCR-003(会員N+1)
SCR-003(会員N) --前へ/めくる--> SCR-003(会員N-1) または SCR-002(目次、N=1の場合)
SCR-003(任意) --「目次へ戻る」クリック--> SCR-002(目次)
```

|---|---|
| ページ端クリック/ドラッグでめくる | FR-SYS-004 |
| 「次へ」「前へ」ボタン | FR-SYS-005 |
| キーボード左右矢印 | FR-SYS-006 |
| 目次から会員名クリック | FR-SYS-007 |
| 全画面ボタン | FR-SYS-009 |

## 9. データ要件

### 9.1 データモデル（DATA-001: Member）

| フィールド | 型 | 必須 | 説明 | 制約 |
|---|---|---|---|---|
| `id` | string | 必須 | 会員の一意識別子 | サイト全体で重複不可、英数字とハイフンのみ |
| `name` | string | 必須 | 会員名 | 1〜100文字 |
| `bio` | string | 任意（既定は空文字） | 紹介文 | 0〜1000文字 |
| `links` | array<Link> | 任意（既定は空配列） | 外部リンク一覧 | 0件以上 |

### 9.2 データモデル（DATA-002: Link）

| フィールド | 型 | 必須 | 説明 | 制約 |
|---|---|---|---|---|
| `label` | string | 必須 | リンクの表示名 | 1〜50文字 |
| `url` | string | 必須 | リンク先URL | `http://` または `https://` で開始必須 |

### 9.3 データライフサイクル

- 作成・更新: `src/data/members.json` をリポジトリ上で直接編集（Phase 0確定方針）
- 保存: リポジトリにコミットされたJSONファイルのみ。データベースは使用しない
- バックアップ: Gitの変更履歴がそのままバックアップとして機能する
- 削除: JSONから該当レコードを削除しビルドし直す（論理削除の概念なし）

## 12. 制約条件

## 14. リスク・変更管理

詳細は [risk_register.md](./risk_register.md) を参照。変更管理プロセス: 要件変更が発生した場合、該当FR/NFRの「変更履歴」欄に日付・変更者・変更内容を追記し、接続先のACC-*/TC-*/RISK-*への影響を確認した上で状態を「要確認」→修正後「Approved」に戻す。エスカレーション基準: Must要件のEvidence根拠が信頼度A/Bで確保できない場合、または破壊的操作（本プロジェクトには存在しない）が発生する変更の場合は社長へエスカレーションする。
